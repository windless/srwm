## Context

本项目是一个全新的 Godot 4.6 移动端游戏项目，旨在复刻《超级机器人大战》的核心战棋玩法。当前项目没有任何实现代码，需要从零构建整个游戏架构。

技术环境：
- **引擎**: Godot 4.6
- **平台**: 移动端 (Mobile)
- **物理**: Jolt Physics (3D)
- **渲染**: Mobile 渲染器

约束条件：
- 需要针对移动设备优化性能
- 战棋游戏的核心循环需要清晰的数据流
- UI 需要适配移动端触控操作

## Goals / Non-Goals

**Goals:**
- 建立可扩展的战棋游戏核心架构
- 实现网格地图系统与单位移动逻辑
- 构建回合制战斗流程
- 设计清晰的数据模型（单位、武器、地形）
- 实现基础的战斗演出效果

**Non-Goals:**
- 本阶段不实现完整剧情系统
- 不实现机甲升级/改造深度系统
- 不实现多关卡/战役系统
- 不实现存档/加载系统
- 不实现多语言支持

## Decisions

### 决策 1: 场景架构

**选择**: 采用单一战场场景 + 子场景组合模式

**理由**:
- 战棋游戏主要在单一战场进行，场景切换频率低
- 子场景组合便于模块化开发和测试
- 避免过度场景切换带来的加载开销

**备选方案**:
- 多场景切换（每次战斗加载新场景）→ 拒绝：加载时间长，不适合战棋游戏节奏

### 决策 2: 地图实现方式

**选择**: 使用 Godot GridMap 节点 + 自定义逻辑层

**理由**:
- GridMap 提供网格定位和碰撞检测基础
- 自定义逻辑层处理地形属性和移动规则
- 支持地图编辑器预编辑和运行时动态生成

**备选方案**:
- 纯 TileMap（2D）→ 拒绝：项目为 3D 设置
- 自定义节点系统 → 拒绝：GridMap 提供足够的网格基础

### 册策 3: 单位数据管理

**选择**: 使用 Resource 文件定义单位数据，运行时实例化

**理由**:
- Resource 文件便于编辑和版本控制
- 实例化支持单位状态修改（位置、HP、等级）
- 符合 Godot 数据管理的最佳实践

**备选方案**:
- JSON 外部文件 → 拒绝：需要额外解析逻辑
- 纯代码定义 → 拒绝：难以维护大量单位数据

### 决策 4: 战斗流程控制

**选择**: 使用独立的 BattleController 单例管理战斗流程

**理由**:
- 战斗流程涉及多个系统协调（地图、单位、UI）
- 单例模式便于跨系统访问和状态同步
- 清晰的回合/阶段管理

**备选方案**:
- 分布式状态管理 → 拒绝：复杂度高，调试困难
- 每个单位自主行动 → 拒绝：难以实现回合同步

### 决策 5: UI 系统架构

**选择**: 使用 Control 节点树 + 信号驱动更新

**理由**:
- Control 节点适合移动端 UI
- 信号驱动避免频繁的主动检查
- 便于 UI 组件复用

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| GridMap 性能在大地图时下降 | 限制地图尺寸，实现视口裁剪 |
| 移动端触控精度问题 | 设计足够大的点击区域，支持长按辅助选择 |
| 战斗演出帧率影响 | 使用简化动画，战斗计算与演出解耦 |
| 单位数据膨胀 | 实现数据热加载，分阶段引入单位 |
