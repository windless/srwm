## Context

当前项目已有单位数据层（UnitData Resource 定义模板，UnitInstance 处理运行时状态），需要新增视觉呈现层来实现机甲单位在战场上的渲染和动画。项目使用 Godot 4.6 和 Mobile 渲染器，需要考虑移动端性能优化。

技术约束：
- 项目为 2D 游戏，坐标定位由外部系统处理（如 GridMapSystem）
- 移动端性能限制，需要控制精灵分辨率和动画帧数
- 战棋游戏视角为固定俯视角度
- 单位数量有限（通常 10-30 个），需要支持同时渲染
- 需要支持编辑器实时预览（使用 @tool 脚本）

精灵纹理结构：
- 每个 PNG 文件包含多个机甲精灵图（sprite sheet）
- 每个机甲图片尺寸：128 x 128 像素
- 每台机甲包含 4 个方向：南、东、北、西
- 每行包含 8 台机体
- 需要通过 AtlasTexture 从精灵表中裁剪单个机甲的纹理

## Goals / Non-Goals

**Goals:**
- 实现单位在战场上的 2D 渲染（坐标由外部系统设置）
- 支持基础动画状态（入场、起飞、待机、移动、攻击、受击、降落、击毁、离场）
- 支持空中状态（airborne）的 y 轴偏移和阴影位置调整
- 使用 AnimatedSprite2D 管理帧动画播放
- 实现动画与单位运行时状态的同步
- 支持单位朝向调整（四方向选择）

**Non-Goals:**
- 不实现复杂战斗演出动画（单独变更处理）
- 不实现机甲自定义外观系统
- 不实现特效系统（命中闪光、伤害数字等）
- 不实现动态光照和阴影投射（使用静态阴影精灵代替）

## Decisions

### 决策 1: 渲染节点结构

**选择**: UnitRenderer 作为独立场景，使用 Node2D + AnimatedSprite2D + CollisionShape2D

```
UnitRenderer (Node2D)
├─ ShadowSprite (Sprite2D)   # 阴影精灵（可选）
├─ AnimatedSprite2D          # 精灵动画播放
```

**理由**:
- Node2D 作为 2D 定位节点（坐标由外部系统设置）
- ShadowSprite 使用半透明阴影纹理，位于机甲下方
- AnimatedSprite2D 是 Godot 2D 帧动画的标准方案
- 不包含 CollisionShape2D，点击检测由外部系统处理
- 独立场景便于复用和测试

**备选方案**:
- Sprite2D + AnimationPlayer → 可选：如果需要更复杂的动画控制
- 直接继承 UnitInstance → 拒绝：职责分离不清

### 决策 2: 动画状态管理

**选择**: 使用 AnimatedSprite2D 的内置动画名称作为状态标识

**理由**:
- AnimatedSprite2D 支持多个动画名称（enter, takeoff, idle, move, attack, hit, land, destroy, exit）
- 通过 play() 方法切换动画，简单直接
- SpriteFrames 资源集中管理所有动画帧
- 符合 Godot 2D 动画的最佳实践

**备选方案**:
- AnimationPlayer + AnimationTree → 可选：如果需要复杂过渡和混合
- 自定义状态机脚本 → 拒绝：AnimatedSprite2D 已足够满足需求

### 决策 3: 精灵资源策略

**选择**: 使用精灵表（Sprite Sheet）+ AtlasTexture 裁剪

**理由**:
- 精灵表格式：每个 PNG 包含多台机甲，每台 128x128，每台机体包含 4 个方向
- 通过 AtlasTexture 从精灵表裁剪单个机甲的纹理
- SpriteFrames 集中管理所有动画帧（每个方向作为独立动画）
- 每个 UnitData 关联机体在精灵表中的索引位置

**精灵表布局**:
```
PNG 尺寸: (8 * 4 * 128) x (N * 128) = 4096 x (行数 * 128)
每台机甲: 128 x 128 (单个方向)
布局方式: 每行包含 8 台机体，每台机体 4 方向水平排列
计算公式: row = unit_index / 8, col_start = (unit_index % 8) * 4
         x = (col_start + direction) * 128, y = row * 128
方向对应: 南=0, 东=1, 北=2, 西=3

示例:
  Unit Index 0: row=0, col_start=0
    方向南: x=0*128=0,     y=0
    方向东: x=1*128=128,   y=0
    方向北: x=2*128=256,   y=0
    方向西: x=3*128=384,   y=0

  Unit Index 1: row=0, col_start=4
    方向南: x=4*128=512,   y=0
    方向东: x=5*128=640,   y=0
    方向北: x=6*128=768,   y=0
    方向西: x=7*128=896,   y=0

  Unit Index 7: row=0, col_start=28
    方向南: x=28*128=3584, y=0

  Unit Index 8: row=1, col_start=0
    方向南: x=0,           y=128
```

**备选方案**:
- 单独 PNG 文件 → 拒绝：增加文件数量，不利于管理
- 运行时动态裁剪 → 可选：如果需要更灵活的资源管理

### 决策 4: 状态同步机制

**选择**: 信号驱动更新

**理由**:
- UnitInstance 发出状态变化信号（如 `action_started`, `action_completed`）
- UnitRenderer 接收信号并触发对应动画
- 解耦数据层和视觉层，便于独立测试
- 符合项目 UI 系统的信号驱动模式

**备选方案**:
- 直接调用 → 拒绝：耦合度高，难以维护
- 定时检查状态 → 拒绝：效率低，可能错过状态变化时机

### 决策 5: 单位朝向实现

**选择**: 四方向独立精灵帧 + 方向索引选择

**理由**:
- 精灵表已提供 4 个方向的独立图片（南、东、北、西）
- 每个方向对应不同的动画帧集合
- 通过选择对应方向的精灵帧实现朝向切换
- 无需翻转，视觉效果更精确

**方向映射**:
```
南 (South) → 方向索引 0，默认朝向（向下）
东 (East)  → 方向索引 1（向右）
北 (North) → 方向索引 2（向上）
西 (West)  → 方向索引 3（向左）
```

**备选方案**:
- flip_h 翻转 → 拒绝：已有四方向精灵，翻转会导致视觉不匹配
- 旋转 Node2D → 拒绝：精灵旋转会导致视觉变形

### 决策 6: 阴影渲染实现

**选择**: 使用独立 Sprite2D 作为静态阴影

**理由**:
- 阴影精灵位于机甲下方，使用半透明灰色纹理
- 简单高效，无需动态光照计算
- 阴影位置可配置偏移（相对机甲位置）
- 符合 2D 战棋游戏的视觉风格

**备选方案**:
- 动态光影系统 → 拒绝：复杂度高，不适合项目阶段
- LightOccluder2D → 可选：如果需要更真实的阴影效果

### 决策 7: 编辑器实时预览

**选择**: 使用 @tool 脚本实现编辑器预览

**理由**:
- @tool 脚本可在编辑器中运行，实时显示渲染效果
- 开发者可在编辑器中调整参数并立即看到变化
- 便于配置精灵索引、朝向等属性
- 提升开发效率和调试体验

**实现要点**:
- 在脚本顶部添加 `@tool` 注解
- 使用 `_process()` 处理编辑器中的更新
- 通过 `Engine.is_editor_hint()` 区分编辑器和运行时逻辑

**备选方案**:
- 仅运行时渲染 → 拒绝：调试效率低，配置困难

### 决策 8: Disabled 状态视觉处理

**选择**: 使用 CanvasItemMaterial 调整灰度效果

**理由**:
- disabled 状态需要将机体显示为灰色
- CanvasItemMaterial 可设置 shader 或 blend mode 实现灰度
- 简单的替代方案：设置 AnimatedSprite2D.modulate 为灰色（如 Color.GRAY）
- 无需额外资源，运行时可动态切换

**备选方案**:
- 独立灰色精灵 → 拒绝：增加资源数量，维护成本高
- ShaderMaterial 灰度效果 → 可选：如果需要更精细的控制

### 决策 9: Airborne 状态渲染

**选择**: 使用 y 轴偏移显示空中机体，阴影位置相应调整

**理由**:
- airborne 状态时，机体 AnimatedSprite2D 的 position.y 偏移（向上）
- 阴影 ShadowSprite 的 position.y 保持不变或向下偏移（产生"悬空"视觉效果）
- 偏移量可配置（如机体 y 偏移 -20，阴影 y 偏移 +10）
- 起飞动画完成后进入 airborne 状态，降落动画完成后退出 airborne 状态

**视觉效果**:
```
地面状态:
┌─────────────────┐
│   [机甲精灵]    │ ← position.y = 0
│                 │
│    [阴影]       │ ← position.y = shadow_offset
└─────────────────┘

空中状态:
┌─────────────────┐
│   [机甲精灵]    │ ← position.y = -20 (向上偏移)
│                 │
│                 │
│    [阴影]       │ ← position.y = shadow_offset + 10 (向下偏移)
└─────────────────┘
```

**动画流程**:
- takeoff → airborne → (idle/move/attack 状态在空中) → land → ground

**备选方案**:
- 使用缩放表示高度 → 拒绝：视觉效果不直观
- 使用透明度变化 → 拒绝：无法准确传达高度信息

## Risks / Trade-offs

| 险险 | 缓解措施 |
|------|----------|
| AnimatedSprite2D 动画过渡不够平滑 | 先接受硬切换，后续可升级为 AnimationTree |
| 移动端精灵内存占用 | 限制精灵分辨率，使用纹理压缩 |
| placeholder 精灵不匹配最终效果 | 设计时考虑替换流程，使用标准化命名 |
| 动画帧资源缺失 | 先用单帧精灵，渐进添加帧动画 |
| @tool 脚本性能影响 | 仅在必要时更新，避免频繁重绘 |

### 决策 10: 测试场景设计

**选择**: 创建独立测试场景，包含信号触发按钮面板

**理由**:
- 测试场景独立于战场场景，便于快速迭代测试
- 按钮面板发射模拟信号，替代 UnitInstance 的信号源
- 每个按钮对应一种状态信号，点击即可触发对应动画
- 可验证所有动画状态（enter, takeoff, idle, move, attack, hit, land, destroy, exit）
- 包含朝向切换按钮和 airborne/disabled 状态切换按钮

**测试面板布局**:
```
┌─────────────────────────────────────┐
│         UnitRenderer 测试场景        │
├─────────────────────────────────────┤
│                                     │
│   [机甲渲染区域]                     │
│                                     │
├─────────────────────────────────────┤
│  动画信号:                           │
│  [deployed] [takeoff] [idle] [move] │
│  [attack] [hit] [land] [destroy]    │
│  [exiting]                          │
├─────────────────────────────────────┤
│  状态切换:                           │
│  [airborne] [disabled]              │
├─────────────────────────────────────┤
│  朝向选择:                           │
│  [南] [东] [北] [西]                 │
└─────────────────────────────────────┘
```

**备选方案**:
- GUT 单元测试 → 可选：补充自动化测试，但视觉验证仍需手动场景
- 直接修改 UnitInstance 测试 → 拒绝：耦合度高，测试场景应独立
