## Context

项目是全新的 Godot 4.6 移动端游戏项目，当前没有任何实现。用户已安装 `gdlint` 工具用于 GDScript lint 检查。需要建立最小的可运行骨架，支持开发者日常工作流程。

技术环境：
- **引擎**: Godot 4.6
- **平台**: macOS (Darwin 25.3.0)
- **Shell**: fish
- **Lint工具**: gdlint（已安装）
- **测试框架**: GUT（待安装）

## Goals / Non-Goals

**Goals:**
- 建立标准目录结构
- 创建可运行的主场景入口
- 实现 check 脚本（lint + 导入检查 + 启动检查）
- 实现 test 脚本（GUT 单元测试运行）
- 实现 run 脚本（启动项目）

**Non-Goals:**
- 不实现任何游戏逻辑
- 不创建复杂的场景架构
- 不配置 CI/CD 流程
- 不编写测试用例（仅准备 test 目录结构）

## Decisions

### 决策 1: 目录结构

**选择**: scenes/ + scripts/ + test/ 三目录结构

**理由**:
- scenes/ 存放所有 Godot 场景文件，符合 Godot惯例
- scripts/ 存放开发者工具脚本，与游戏代码区分
- test/ 存放单元测试，GUT 默认查找路径

**备选方案**:
- 单一根目录 → 拒绝：不利于后续扩展
- addons/ 作为唯一扩展目录 → 拒绝：开发者脚本不适合放插件目录

### 决策 2: 主场景类型

**选择**: 空白 Node 场景

**理由**:
- 最简单，无任何依赖
- 后续开发时可替换为正式的 BattleScene
- 不引入任何临时性的视觉元素

**备选方案**:
- Node2D + Label "Hello World" → 拒绝：临时代码会被删除
- 带脚本的管理器场景 → 拒绝：现在没有内容需要管理

### 决策 3: check 脚本检查层次

**选择**: 三层检查（gdlint → 导入 → 启动）

**理由**:
- gdlint 检查命名、结构、格式（真正的 lint）
- --import 检查资源引用错误
- --headless --quit 检查项目能否正常启动

**备选方案**:
- 仅 gdlint → 拒绝：遗漏运行时错误检查
- 仅 Godot CLI → 拒绝：缺少 lint 检查

### 决策 4: test 脚本依赖

**选择**: 检查 GUT 是否安装，未安装时报错退出

**理由**:
- GUT 需要通过 Asset Library 安装，不是自动依赖
- 明确的错误提示帮助开发者理解缺失的依赖

**备选方案**:
- 自动下载安装 GUT → 拒绝：过于复杂，不尊重用户选择
- 不检查依赖 → 拒绝：错误信息不清晰

### 决策 5: 脚本语言

**选择**: Bash 脚本

**理由**:
- 简单直接，适合 macOS/Linux 环境
- Godot CLI 和 gdlint 都是命令行工具，Bash 是自然接口

**备选方案**:
- Python 脚本 → 拒绝：增加不必要的依赖
- fish 脚本 → 拒绝：Bash 更通用，兼容性更好

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| GUT 未安装时 test.sh 失败 | 脚本检查并提供安装提示 |
| gdlint 配置过于严格 | 使用默认配置，后续可按需调整 |
| 空项目启动检查有警告 | 可接受，等待实际代码后再完善 |
| 脚本没有权限 | 提示用户 chmod +x |
