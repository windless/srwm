## Context

当前项目已完成单位系统（UnitData、WeaponData、PilotData）和单位渲染动画系统的实现。战棋游戏的核心玩法需要在网格地图上进行单位部署、移动和战斗，但项目缺少地图系统。

技术环境：
- **引擎**: Godot 4.6
- **平台**: 移动端 (Mobile)
- **物理**: Jolt Physics (3D)
- **渲染**: Mobile 渲染器

视觉目标：
- 实现类似《超级机器人大战》的经典斜视角（Isometric）2.5D 显示效果
- 使用 TileMap 自定义切片绘制地图
- 支持编辑器中实时显示网格线

约束条件：
- TileMap 提供 2D 网格渲染基础，需要自定义逻辑层处理地形属性
- 斜视角需要特殊的坐标转换（网格坐标 ↔ 屏幕坐标）
- 移动端需要优化地图渲染性能
- 需要与现有的单位渲染系统集成，提供坐标定位和视觉对齐

## Goals / Non-Goals

**Goals:**
- 建立统一的网格坐标系统，用于定位地图上的所有位置
- 定义地形类型和属性，支持移动消耗、防御修正等地形效果
- 实现地图管理节点，支持地图生成、加载和查询
- 实现 2.5D 斜视角显示效果（类似经典 SRW）
- 支持编辑器中实时显示网格线和地图切片
- 为战斗系统提供地形信息查询接口

**Non-Goals:**
- 不实现复杂的地图编辑器功能（仅支持 TileMap 编辑器预览）
- 不实现动态地形变化（如地形破坏）
- 不实现多层地图（如地下层、空中层）
- 不实现地图保存功能
- 不实现真正的 3D 视角（仅模拟斜视角 2.5D）

## Decisions

### 决策 1: 坐标表示方式

**选择**: 使用自定义 GridCoord 结构体（基于 Vector2i）

**理由**:
- Vector2i 提供高效的整数坐标操作
- GridCoord 结构体可添加额外方法（有效性检查、距离计算）
- 与 Godot GridMap 的网格定位概念一致

**备选方案**:
- 使用 Vector3（包含高度）→ 拒绝：当前不需要高度维度，会增加复杂度
- 纯整数变量 x, y → 拒绝：缺乏类型安全性和封装

### 决策 2: 地形数据存储

**选择**: 使用 Resource 文件定义地形模板

**理由**:
- Resource 文件便于在编辑器中配置地形属性
- 支持版本控制和数据热加载
- 与现有的 UnitData、WeaponData 设计风格一致

**备选方案**:
- JSON 外部文件 → 拒绝：需要额外解析逻辑，不符合 Godot 数据管理惯例
- 纯代码枚举 → 拒绝：难以扩展地形类型，不便于配置复杂属性

### 决策 3: 地图管理架构

**选择**: 使用独立 GridMapSystem 节点管理地图状态，配合 TileMapLayer 进行渲染

**理由**:
- GridMapSystem 管理逻辑层（地形属性、坐标查询）
- TileMapLayer 负责视觉渲染（地形切片绘制）
- 分离设计便于后续扩展（如地图加载、地图切换）
- TileMapLayer 是 Godot 4.6 推荐的新 API（TileMap 已弃用）

**备选方案**:
- 将地图逻辑嵌入 BattleController → 拒绝：职责混淆
- 使用旧版 TileMap → 拒绝：Godot 4.6 已弃用，建议使用 TileMapLayer
- 使用全局单例 → 拒绝：地图状态可能需要多次重置

### 决策 4: 斜视角显示实现

**选择**: 使用 TileMap Isometric 模式 + 自定义坐标转换

**理由**:
- Godot TileMap 内置 Isometric 支持
- 斜视角符合《超级机器人大战》经典视觉风格
- 单位渲染系统可复用现有的 UnitRenderer（需适配斜视角坐标）

**备选方案**:
- 正交视角 + 3D 网格 → 拒绝：不符合用户要求的 SRW 风格
- 自定义 Shader 实现 → 拒绝：TileMap 内置方案更简洁

### 决策 5: 网格线可视化

**选择**: 使用 GridVisualization 组件绘制网格线

**理由**:
- 网格线需要在编辑器中实时显示，便于地图设计
- 使用 Godot CanvasItem 绘制（自定义 _draw 方法）
- 支持运行时动态显示/隐藏网格线
- 可扩展支持移动范围高亮、攻击范围可视化

**备选方案**:
- 使用 TileMap 自定义切片绘制网格 → 拒绝：灵活性不足，难以动态控制
- 使用 3D Mesh 绘制 → 拒绝：与 2D 斜视角不兼容

### 决策 6: 地形数据结构

**选择**: 使用二维数组存储每个网格的地形类型引用

**理由**:
- 简单高效的坐标访问（O(1)）
- 内存占用可控
- 便于序列化和反序列化

**备选方案**:
- Dictionary 嵌套 → 拒绝：访问效率较低，调试不便
- GridMap MeshLibrary → 拒绝：MeshLibrary 用于视觉渲染，不适合存储逻辑属性

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 大地图内存占用 | 限制地图尺寸上限（如 50x50），TileMap 支持视口裁剪 |
| 地形数据膨胀 | 使用懒加载策略，按关卡加载地形数据 |
| 移动端触控精度 | 提供足够大的网格选择区域，支持网格边界高亮 |
| 斜视角坐标转换复杂 | 封装 GridCoord.to_screen() 和 from_screen() 方法 |
| TileMap 与逻辑层同步 | GridMapSystem 监听 TileMap 变化事件，保持数据一致性 |
| 单位渲染与斜视角对齐 | UnitRenderer 使用 GridMapSystem 提供的屏幕坐标转换 |
